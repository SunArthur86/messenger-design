# 国内其他消息 APP 技术架构调研

> 调研日期：2026-06-03
> 覆盖：钉钉、企业微信、抖音私信、微博、网易云信/融云/环信（第三方 IM PaaS）

---

## 一、钉钉（DingTalk）— DTIM 架构

钉钉拥有 2100 万+组织、5 亿+注册用户。其 IM 服务 DTIM（DingTalk IM）是首次对外深度公开技术设计的 ToB IM 系统。

### 1.1 三大核心服务

| 服务 | 职责 |
|------|------|
| **消息服务** | IM 核心模型 + 开放 API：消息发送、单聊群聊、群元信息、历史拉取、已读通知、跨域转发 |
| **同步服务** | 端到端数据同步：长连接实时推送 + 多端一致 + 离线补推 |
| **通知服务** | 三方通道推送：厂商 Push（APNs/华为/小米等）+ 透传 |

### 1.2 读写混合存储模型

钉钉采用**读扩散 + 写扩散混合方案**，是三者中唯一公开混合方案的：

- **conversation_message 表**（读扩散）：群消息只存一份，所有成员共享读取
- **message_inbox 表**（写扩散）：每个用户独立收件箱，存两种特殊消息：
  - **定向消息 P（Private）**：只对特定人可见（如红包领取状态）
  - **状态转换 NP（Normal→Private）**：用户删除自己的消息副本等个性化操作
- 用户拉取时从两表合并，按时间排序输出

> **核心洞见**：6% 的用户贡献 50% 消息量，写扩散会带来 1:30 的扩散比，纯写扩散存储成本过高。

### 1.3 推优先同步模型（PPM）

钉钉放弃了推拉结合方案，采用**推优先模型（Preferentially-Push Model）**：

- 服务端主动推送，延迟毫秒级
- 支持消息**可重入**（客户端按 msgId 去重）
- 支持消息**按时间戳排序**（防乱序）
- **Rebase 机制**：消息堆积超阈值时，发送 Rebase 事件，客户端跳到最新消息，历史按需回溯

### 1.4 存储优化：KV 分离

- 底层使用表格存储（LSM 架构），面临读写放大问题
- 借鉴 WiscKey 思想改进：相同前缀的多个 Key 聚合判断，Value 超阈值则打包分离到 value-block
- Minor Compaction 阶段将 70% 消息放入 value 文件，大幅降低写放大

### 1.5 单元化架构

- **会话维度**划分单元（非用户维度），消息处理封闭在单个单元内
- 全局路由服务（RoutingService）：动态调度流量，支持企业级弹性扩展
- 单元降级：故障单元流量自动切换，用户无感知

### 1.6 大群优化

- 大群红点/Lastmsg：发消息时不更新，拉首屏时实时算，高峰减少 99.99% 存储操作
- 已读请求合并：1 分钟内所有请求合并为 1 次，扩散量减少 96%
- 消息削峰：低优先级消息保证 1 小时内处理

## 二、企业微信（WeCom）— ToB IM 设计

企业微信由微信事业群（WXG）开发，继承了微信架构基因，但面临 ToB 独特挑战。

### 2.1 架构分层

| 层 | 组件 | 说明 |
|----|------|------|
| 接入层 | wwproxy | 长连接优先，失败降级短连接 |
| CGI 层 | http 服务 | session 校验 + 加解密 |
| 逻辑层 | hikit RPC | 微服务集群，TCP 短连通信 |
| 存储层 | msgkv + SeqSvr | 基于 LevelDB 自研 + 序列号生成 |

### 2.2 消息扩散写 + 单序列号

企业微信采用**纯扩散写**方案：

- 每条消息存 N 份（N = 群成员数），每个用户独立消息流
- 每个用户**只需一个 seq**，客户端保存最大 seq，增量拉取
- 冷热分离：冷数据存 SATA 盘降低成本

### 2.3 万人群优化三板斧

| 优化 | 说明 |
|------|------|
| **并发限制** | 单群分发并发上限 5，万人分发约 16s 完成 |
| **合并插入** | 队列积压时多条消息合并为一个请求，吞吐量成倍增加（最高 20 条合并） |
| **业务降级** | 万人群控制消息直接丢弃不分发，避免千万级扩散 |

### 2.4 柔性可用策略

- **"先成功后重试"**：逻辑层 hold 住失败请求，返回前端成功，后台异步重试
- 指数退避重试，最多 hold 半小时
- 复杂鉴权（好友/企业/集团/圈子关系）失败时幂等重试

### 2.5 已读回执设计

采用 **referid 指向方案**：
- 接收方已读时，在其消息流插入一条新消息指向原消息
- 发送方合并所有已读状态（通过 MQ 串行化避免覆盖写）
- 合并处理节省 44% 写入量

### 2.6 微信互通（ImUnion）

- 两个异构 IM 系统打通，鉴权由 ImUnion 外部系统处理
- 全异步化解耦：外部系统故障不影响主消息链路
- 先返回成功，异步失败则回调客户端出红点

## 三、抖音私信

### 3.1 短视频平台消息特点

- 消息与内容强关联：视频分享、评论回复、直播互动引流到私信
- 明星/KOL 场景：单账号可能面临百万粉丝同时私信
- 场景多样化：1v1 聊天、粉丝群、直播群聊、商家客服

### 3.2 技术架构要点

| 维度 | 方案 |
|------|------|
| 消息同步 | 类飞书 Timeline 模型 + 推拉结合 |
| 连接方案 | WebSocket 长连接 + 厂商 Push |
| 存储方案 | 字节跳动自研分布式存储（类似 HBase） |
| 热点处理 | KOL 收件箱限流 + 消息聚合 + 异步处理 |
| 内容安全 | 先审后发/先发后审分级策略 |

### 3.3 特殊场景优化

- **陌生人私信**：非好友场景的消息投递，需经过反骚扰/反垃圾过滤
- **商家客服**：客服排队分配 + 消息路由 + 会话结束超时
- **直播弹幕→私信**：高并发场景下消息降级（限频/聚合/丢弃旧消息）

## 四、微博私信

### 4.1 社交媒体消息特点

- 明星粉丝场景：单账号可能收到海量私信（类似 KOL 场景）
- 消息分级：普通私信 vs 认证用户私信 vs 商业私信
- 强内容审核：政治敏感/广告/骚扰过滤

### 4.2 技术要点

| 维度 | 方案 |
|------|------|
| 消息模型 | 读扩散为主（明星场景写扩散不可行） |
| 存储方案 | Redis 缓存 + MySQL 持久化 |
| 限流策略 | 按发送频率/接收量多维度限流 |
| 消息过滤 | 多级过滤管道（关键词→模型→人工审核） |

## 五、第三方 IM PaaS 对比

### 5.1 市场格局

| 服务商 | 背景 | 月消息量 | 核心优势 |
|--------|------|---------|---------|
| **网易云信** | 网易 20 年 IM 积累 | 2000 亿+ | 稳定性、音视频融合、大厂技术体系 |
| **融云** | 前中国移动飞信团队 | 1500 亿+ | 全球化布局、IM+RTC+AI 融合 |
| **环信** | 硅谷团队 | 1000 亿+ | 高并发、全球化节点、客服场景强 |
| **腾讯云 IM** | QQ/微信技术输出 | — | 微信基因、低延迟、小程序生态 |
| **极光 IM** | 推送起家 | — | 推送+IM 一体化、集成成本低 |

### 5.2 技术架构共性

```
┌─────────────────────────────────────────┐
│            第三方 IM PaaS 架构           │
├─────────────────────────────────────────┤
│  SDK (iOS/Android/Web/Flutter/RN)       │
│    ↓ Protobuf / MQTT / WebSocket        │
│  接入网关 (负载均衡/协议适配/限流)        │
│    ↓                                    │
│  消息路由 (多租户隔离/消息分发)           │
│    ↓                                    │
│  存储层 (消息KV/会话/用户/关系链)        │
│    ↓                                    │
│  多租户数据隔离 (租户ID分片)             │
└─────────────────────────────────────────┘
```

### 5.3 核心技术差异

| 维度 | 网易云信 | 融云 | 环信 |
|------|---------|------|------|
| 连接协议 | TCP 自定义 + WebSocket | MQTT + WebSocket | MQTT + WebSocket |
| 消息存储 | 自研分布式存储 | MongoDB + Redis | HBase + Redis |
| 多租户隔离 | AppKey 分库 | AppKey 路由 | AppKey 命名空间 |
| 全球化 | 专线加速 | 海外 30+ 节点 | 海外 PoP 节点 |
| 特色 | 音视频低延迟 | 万人大群方案 | 智能客服集成 |
| SDK 平台 | 全平台 + 小程序 | 全平台 + 小程序 | 全平台 + 小程序 |

### 5.4 选型建议

| 场景 | 推荐 | 理由 |
|------|------|------|
| 社交 APP | 融云/网易云信 | 高并发、消息可靠性 |
| 企业协作 | 腾讯云 IM/网易云信 | 组织架构、权限管理 |
| 客服系统 | 环信 | 智能客服、排队分配 |
| 游戏聊天 | 网易云信 | 低延迟、弹幕场景 |
| 快速集成 | 极光 IM | 推送+IM 一体化 |
| 海外业务 | 融云 | 全球化节点最多 |

## 七、关键洞见

### 7.1 读写扩散的选择矩阵

```
                    写扩散成本可接受
                    ┌──────┬──────┐
                    │  是  │  否  │
              ┌─────┼──────┼──────┤
  需要个性化  │ 是  │ 写扩散│ 混合  │
  (已读/删除) │     │      │(钉钉)│
              ├─────┼──────┼──────┤
              │ 否  │ 写扩散│ 读扩散│
              │     │(企微)│(微博)│
              └─────┴──────┴──────┘
```

### 7.2 企业 IM vs 社交 IM 的核心差异

| 维度 | 企业 IM | 社交 IM |
|------|---------|---------|
| 消息可靠性 | 极高（99.995%+） | 高 |
| 已读回执 | 必须 | 可选（隐私） |
| 消息存储 | 长期漫游（180天+） | 短期为主 |
| 群规模 | 万人级 | 百人~千人级 |
| 内容审核 | 企业安全 | 公共安全 |
| 柔性策略 | 先成功后重试 | 失败即反馈 |

### 7.3 第三方 IM vs 自研的决策

| 条件 | 选第三方 | 选自研 |
|------|---------|--------|
| 团队规模 | < 10 人 IM 专项 | > 30 人 |
| 上线时间 | < 3 个月 | > 6 个月 |
| 定制需求 | 标准功能 | 深度定制 |
| 成本预算 | 低（按量付费） | 高（基础设施投入） |
| 数据安全 | 可接受第三方托管 | 必须自控 |
