# 飞书、微信、QQ 消息系统技术架构调研

> 调研日期：2026-06-03
> 调研范围：飞书（Lark）、微信、QQ 三款国民级 IM 的核心技术架构差异

---

## 一、飞书（Lark）技术架构特点

飞书是字节跳动旗下的企业协作平台，定位于企业办公场景，其 IM 系统需同时满足实时通信和企业级协同需求。

### 1.1 消息同步机制

- **Timeline（时间轴）模型**：飞书 IM 采用基于 Timeline 的消息同步模型，每个用户的会话维护一条独立的时间轴，包含消息体、已读位点、最大位点等基本模块
- **推拉结合**：在线用户通过长连接实时推送新消息通知，离线用户上线后通过增量拉取同步未读消息
- **消息有序性保障**：服务端为每条消息分配全局有序的消息 ID，确保多端消息顺序一致

### 1.2 长连接方案

- **WebSocket 长连接**：飞书客户端通过 WebSocket 协议维持与服务端的长连接，实现毫秒级消息推送
- **开放平台长连接模式**：第三方应用可通过飞书 SDK（支持 Golang/Python/Java/Node.js）建立长连接接收事件回调，无需公网 IP 或域名
- **心跳保活**：客户端定期发送心跳包，服务端通过心跳上报参数管理连接状态

### 1.3 离线消息与多端同步

- **云端存储优先**：所有消息存储在云端，支持多端同步、消息漫游和在线检索
- **位点同步机制**：各端维护本地已同步的最大位点（类似微信的 sequence），上线后增量拉取差量消息
- **多端已读同步**：服务端记录每个用户在各会话中的已读位点，多端登录时已读状态实时同步

### 1.4 已读回执

- **企业级已读回执**：飞书作为企业 IM，原生支持消息已读回执功能，单聊和群聊均可查看已读/未读成员列表
- **已读事件推送**：通过开放平台事件订阅，第三方应用可接收 `消息已读` 事件回调
- **已读位点头像展示**：群聊中展示"已读到此处"的成员头像，基于已读位点聚合实现

### 1.5 消息存储模型

- **后端技术栈**：飞书后端采用 Go 语言微服务架构，引入 CloudWeGo/Kitex 框架实现高性能 RPC 通信
- **平台化架构**：管理后台采用平台化改造，通过 Kitex 泛化调用将单体应用拆分为可扩展的微服务集群
- **Timeline 模型存储**：消息存储以 Timeline 为核心抽象，支持全量消息云端存储和索引

### 1.6 消息类型丰富度

- 支持文本、富文本、图片、视频、文件、消息卡片（Interactive Card）等多种消息类型
- 消息卡片支持交互按钮，将 IM 消息扩展为轻量级工作流入口
- 群聊支持 thread（话题）模式，消息按话题分组，适合企业场景下的异步讨论

## 二、微信技术架构特点

微信诞生于 QQMail 团队，初始架构带有浓厚的邮箱"存储转发"基因。经过 13 年演进，已发展为支撑 13.4 亿 DAU 的超级 IM 系统。

### 2.1 SYNC 协议（核心同步机制）

- **参考 ActiveSync 设计**：微信自研 SYNC 协议，将"收发消息"抽象为"状态同步"过程
- **基于状态同步**：假定终端和服务器之间存在状态差异，收消息本质是同步服务器端最新状态（消息）的过程
- **统一交互模型**：无论是收取消息还是获取好友状态更新，都统一为同一个同步流程
- **最小化传输**：通过增量差值同步，只传输变化的数据，在 GPRS 等弱网环境下也能流畅使用

### 2.2 消息收发架构

- **三层架构**：接入层（ConnectSvr）→ 逻辑层（SendSvr/ReceiveSvr）→ 存储层（MsgStore）
- **发送流程**：手机 A → ConnectSvr → SendSvr（反垃圾/黑名单等逻辑）→ MsgStore → PushSvr → 手机 B
- **双通道推送**：PushSvr 同时通过长连接推送 + 第三方 Push（APNs 等）通知接收方
- **消息实时性**：保证 100ms 级别的端到端消息延迟

### 2.3 Sequence 序列号机制

- **用户级序列号空间**：每个用户拥有 42 亿（1 ~ UINT_MAX）的独立 sequence 空间，连续递增分配
- **消息防丢失**：服务端存储每个用户已分配的最大 sequence，客户端存储已收取的最大 sequence，通过差异增量拉取保证不丢消息
- **SeqSvr 段分配**：序列号生成服务采用段分配策略，每次从存储层分配步长 10000 的段，段内分配纯内存操作，延迟极低
- **多端登录兼容**：服务器记录已确认下发的最大 sequence，不同终端轮流登录不会收到重复消息

### 2.4 信箱模型（Mailbox Model）

- **按用户存储**：每个用户拥有独立信箱，消息分别写入发送方和接收方信箱
- **天然支持多端同步**：各信箱独立维护 sequence，多端同步无需额外机制
- **离线消息天然支持**：上线后直接拉取信箱增量，无需单独的离线消息队列
- **群消息写放大**：群消息需写 N 份（N 为群成员数），通过内容去重（消息体只存一份，信箱存引用）、批量写入、异步写入缓解

### 2.5 短连接 + 长连接混合方案

- **Mars 网络库**：微信开源的跨平台网络库，包含 STN（信令传输网络）、SDT（网络诊断）、Xlog（高性能日志）
- **复合连接策略**：优先建立长连接（自定义二进制协议），5 秒内未建立则并行尝试 HTTPS 短连接，"竞速"取最快
- **智能心跳**：自适应心跳算法（60s ~ 300s），逐步增大间隔直到发现失败阈值后回退，适配不同运营商 NAT 超时
- **三级 IP 获取**：本地缓存 → HTTP DNS（绕过运营商劫持）→ 系统 DNS 兜底

### 2.6 朋友圈读扩散架构

- **读扩散模式（Fan-out on Read）**：发布时只写自己的发件箱，读取时实时聚合所有好友的内容
- **选型原因**：微信双向好友关系，好友上限 5000，不存在百万粉丝场景，读扩散避免写入放大
- **多级缓存**：客户端 SQLite 缓存 + 更新通知（红点）+ 增量加载 + 服务端缓存层

### 2.7 容灾与存储架构

- **SET 化（单元化）架构**：用户按 ID 哈希划分到不同 SET，每个 SET 包含完整服务和数据
- **PaxosStore**：基于 Paxos 协议的分布式存储，三副本部署，保证强一致性
- **灰度发布**：每天支撑 20+ 后台变更，通过灰度、灰度、再灰度策略保证稳定性

## 三、QQ 技术架构特点

QQ 是中国互联网存活最久的国民级 IM，历经 25 年 6 次关键架构跃迁，从万级到亿级在线的技术演进是 IM 领域的完整参考模型。

### 3.1 历史变迁：从 UDP 到 TCP

| 阶段 | 时间 | 在线规模 | 核心架构 |
|------|------|---------|---------|
| 初创求生期 | 1999-2001 | 90人→百万 | UDP 单体架构，应用层实现可靠性 |
| 百万在线攻坚 | 2002-2004 | 百万→3000万 | 三层拆分（接入+逻辑+存储），引入 MySQL 主从 |
| 生态爆炸期 | 2005-2010 | 1亿→1.2亿 | 全内存存储为主，分布式缓存+对象存储 |
| 移动转型期 | 2011-2014 | 1亿→2亿 | 动态心跳弱网优化，移动专用接入网 |
| 云原生期 | 2015-2021 | 稳定1.4亿 | 微服务拆分，K8s 容器化，全面上云 |
| NT 架构 | 2022-至今 | 1.4亿+ | 跨平台 C++ 内核 + Electron 桌面端 |

### 3.2 UDP 协议选型的历史原因

- **带宽稀缺年代**：1999 年服务器和带宽极其昂贵，UDP 头部仅 8 字节，比 TCP 省约 30% 带宽
- **应用层可靠性**：QQ 在 UDP 之上自建可靠性机制（超时重传、校验和验证、消息有序化、消息回执）
- **UDP→TCP 演进**：随着带宽成本下降和移动网络普及，QQ 逐步引入 TCP 长连接，最终形成 UDP+TCP 混合模式
- **现代架构**：登录成功后均通过 TCP 连接保持在线状态，UDP 作为辅助传输通道

### 3.3 群消息分发机制

- **写扩散模型**：群消息向所有成员推送，早期采用同步处理，后引入消息队列异步处理扩散逻辑
- **内存优先策略**：偷菜洪峰时期验证了全内存架构的可行性，群消息核心逻辑全内存运行
- **批量投递优化**：消息批量投递到群成员，减少单条投递带来的网络开销和数据库压力
- **超大群架构**：从最初 10 人群上限，演进到支持百万人超大群聊

### 3.4 在线状态（Presence）系统

- **统一状态中心**：搭建全局在线状态服务，管理用户在各终端的在线/离线状态
- **多端状态同步**：PC + 手机 + Web 多端登录时，通过会话原子更新确保多端状态实时一致
- **状态推送机制**：好友状态变更通过接入层推送至所有关注该用户在线状态的客户端
- **动态心跳适配**：WiFi 环境 1 分钟/次心跳，2G 环境 5 分钟/次心跳，平衡连接稳定性和资源消耗

### 3.5 NT 架构（跨平台统一）

- **C++ 跨平台内核**：将 IM 核心业务逻辑（消息、关系链、富媒体、音视频）统一下沉到 C++ 层
- **统一消息 ID**：解决历史版本无统一消息 ID 的问题，为多端消息同步打下基础
- **统一消息格式**：各平台约 200 种消息类型精简统一，消除历史包袱
- **新客户端 DB**：设计跨平台统一数据库，支持断点续导、用户分级迁移，确保消息不丢失不重复
- **MVI 架构**：聊天窗口采用单向数据流 MVI 架构 + 预加载 + 异步渲染，实现丝滑体验

### 3.6 微服务与云原生

- **巨石拆分为近百微服务**：按业务域独立部署迭代，发布周期从周级缩短至天级
- **K8s 弹性伸缩**：支撑春节红包等极端并发场景，Pod 数量快速扩容 10 倍
- **全链路可观测**：监控告警、日志分析、链路追踪一体化，故障定位从小时级降至分钟级

## 四、三者核心差异对比

| 维度 | 飞书（Lark） | 微信 | QQ |
|------|-------------|------|-----|
| **产品定位** | 企业协作平台 | 全民社交+生活 | 全民社交+娱乐 |
| **用户规模** | 千万级企业用户 | 13.4亿 DAU | 亿级 MAU |
| **消息同步协议** | Timeline 模型 + 推拉结合 | SYNC 协议（类 ActiveSync）+ Diff-Sync | 传统推模式 + 历史拉取 |
| **消息 ID 设计** | 服务端生成全局有序 ID | Sequence 序列号（用户级递增） | 历史无统一 ID，NT 架构后统一 |
| **消息存储模型** | Timeline 时间轴模型 | 信箱模型（Mailbox Model） | 从文件存储到 DB，NT 后统一 DB |
| **群消息分发** | 写扩散 + Timeline | 写扩散 + 消息引用去重 | 写扩散 + 异步队列 |
| **连接方案** | WebSocket 长连接 | Mars：长连接 + 短连接竞速 | UDP→TCP→长连接演进 |
| **离线消息** | 云端存储 + 增量拉取 | 信箱增量（sequence 差值） | 服务端暂存 + 上线拉取 |
| **已读回执** | 原生支持（企业级） | 不支持（社交隐私保护） | 支持 |
| **多端同步** | 云端优先，多端实时同步 | 信箱 sequence 天然支持 | NT 架构统一数据同步层 |
| **弱网优化** | WebSocket 重连机制 | Mars 智能心跳 + 竞速连接 | 动态心跳 + 弱网补偿 |
| **后端技术栈** | Go 微服务 + CloudWeGo/Kitex | C++ 核心服务 + 自研组件 | C++ 内核 + 微服务 + K8s |
| **存储引擎** | 分布式存储（推测自研） | PaxosStore（Paxos 协议） | MySQL→内存→云原生存储 |
| **容灾方案** | 微服务化 + 多副本 | SET 化 + PaxosStore | 多地多中心 + K8s 弹性 |
| **信息流架构** | 不适用（企业场景） | 朋友圈：读扩散 | QQ空间：写扩散+读扩散混合 |
| **核心架构特色** | Timeline 模型 + 开放平台 | SYNC 协议 + 信箱模型 + Mars | 从 UDP 到 NT 的完整演进 |

## 五、关键差异深度分析

### 5.1 消息同步范式

- **微信**：SYNC 协议将所有数据同步抽象为"状态同步"，消息、好友状态、朋友圈更新统一走同一通道，交互模式极简——服务端只推一个轻量通知，客户端按需拉取。这种设计在移动网络环境下流量极省、可靠性高。
- **飞书**：采用更现代的 Timeline 模型，每个会话维护独立时间轴，天然支持多端同步和消息漫游。企业场景对消息可靠性和多端一致性要求更高。
- **QQ**：历史包袱最重，早期无统一消息 ID 和存储方案，NT 架构通过全链路重写解决根本问题。

### 5.2 消息可靠性保障

- **微信**：Sequence 机制 + 推拉结合，即使推送通知丢失，客户端也能通过 sequence 差值正确拉取未收消息。分页拉取避免弱网下大数据传输超时。
- **飞书**：云端存储 + Timeline 位点同步，企业场景要求消息零丢失。
- **QQ**：从 UDP 时代的应用层可靠性（超时重传+回执），演进到 TCP 长连接的天然可靠传输。

### 5.3 架构演进哲学

- **微信**："大道至简"——SYNC 协议、信箱模型、Mars 网络库都是用简单方案解决复杂问题的典范。"前轻后重"——功能点后移到服务端实现，充分发挥后台快速变更能力。
- **飞书**：作为后来者，站在前人肩膀上，采用更现代的技术栈（Go + 微服务 + Timeline 模型），没有历史包袱。
- **QQ**："业务驱动技术"——25 年间每一步架构跃迁都精准对应业务痛点，从单体到分布式到云原生到跨平台，是 IM 系统架构演进的完整教科书。
