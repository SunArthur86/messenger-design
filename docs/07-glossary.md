# IM 系统术语表

> 即时通讯（Instant Messaging）领域核心术语速查手册，涵盖消息同步、存储、安全、网络等各方面。

---

## A

### 1. ACK（确认应答）

**英文**：Acknowledgment

**一句话解释**：接收方收到消息后向发送方返回的确认信号。

**详细说明**：在 IM 系统中，ACK 机制用于保证消息的可靠投递。客户端收到消息后向服务端发送 ACK，服务端据此标记消息已送达。如果超时未收到 ACK，服务端会触发重传。常见的 ACK 层级包括：Server ACK（服务端确认收到）、Client ACK（目标客户端确认收到）、Read ACK（已读确认）。

### 2. APNs（Apple Push Notification Service）

**英文**：Apple Push Notification Service

**一句话解释**：苹果提供的远程推送服务，用于 iOS 应用在后台时唤醒客户端接收消息。

**详细说明**：IM 应用在 iOS 后台被系统挂起后无法维持长连接，需要依赖 APNs 将消息推送到用户设备。客户端收到推送后可以展示通知，并在用户点击时唤醒应用拉取最新消息。APNs 是 iOS IM 应用的离线推送基础。

### 3. Async（异步消息处理）

**英文**：Asynchronous Messaging

**一句话解释**：消息发送方不需要等待接收方立即处理，通过队列等方式解耦的通信模式。

**详细说明**：IM 系统广泛采用异步处理，如异步持久化、异步推送、异步已读回执等，以提升吞吐量和降低延迟。

---

## B

### 4. BEAM 虚拟机

**英文**：BEAM (Bogdan's Erlang Abstract Machine)

**一句话解释**：Erlang/Elixir 语言的运行时虚拟机，原生支持轻量级进程和高并发。

**详细说明**：BEAM 通过抢占式调度器在多核 CPU 上调度数百万个轻量级进程，每个进程内存仅几百字节。WhatsApp 利用 Erlang/BEAM 用少量服务器支撑了数十亿并发连接，是 IM 高并发后端的经典案例。

### 5. 班级消息（Class Message）

**英文**：Groupcast

**一句话解释**：面向特定群组的定向广播消息。

**详细说明**：常见于教育、企业场景中，向特定组织单元（如班级、部门）批量发送的通知消息，通常采用写扩散或读扩散优化分发效率。

---

## C

### 6. CDN（内容分发网络）

**英文**：Content Delivery Network

**一句话解释**：通过分布式节点加速内容分发的网络架构。

**详细说明**：IM 系统中用于加速图片、视频、文件等富媒体消息的分发，降低中心服务器带宽压力并减少用户下载延迟。

### 7. Cell-Based Architecture（单元化架构）

**英文**：Cell-Based Architecture

**一句话解释**：将系统按业务维度切分为独立自治的"单元"，每个单元内包含完整的服务栈。

**详细说明**：钉钉、Slack 等大型 IM 采用单元化架构实现故障隔离和水平扩展。每个 Cell 拥有独立的接入层、逻辑层和存储层，Cell 之间通过路由网关协调。当某个 Cell 故障时，仅影响该 Cell 内的用户，不会波及全局。

### 8. Connection Pool（连接池）

**英文**：Connection Pool

**一句话解释**：预创建并复用网络连接的池化技术。

**详细说明**：在 IM 服务端，连接池用于管理与数据库、缓存、消息队列等下游组件的连接，避免频繁创建/销毁连接的开销，提升系统稳定性。

---

## D

### 9. DTIM（钉钉即时消息架构）

**英文**：DingTalk Instant Messaging

**一句话解释**：钉钉自研的企业级 IM 架构体系。

**详细说明**：DTIM 采用推优先同步模型（PPM）、读写混合存储、KV 分离存储优化、单元化架构等核心技术，支撑钉钉数亿用户的企业沟通需求。其设计理念强调高可靠、强一致和柔性可用。

### 10. 单序列号（Single Sequence Number）

**英文**：Single Sequence Number

**一句话解释**：每个用户拥有全局唯一递增的序列号，用于消息同步。

**详细说明**：企业微信等采用单序列号方案，用户的所有消息（单聊、群聊）共享一个递增序列号。客户端通过比对本地上和服务端的序列号差异来增量拉取新消息，简化了同步逻辑。

### 11. 读扩散（Read Diffusion / Fan-out on Read）

**英文**：Read Diffusion / Fan-out on Read

**一句话解释**：消息只存一份在发件箱，收件人读取时再聚合拉取。

**详细说明**：读扩散模式下，发送者消息写入自己的 Timeline，接收方需要查询所有关注者的 Timeline 并合并排序。优点是写入开销小（只写一次），适合写多读少的场景（如微信朋友圈）。缺点是读取时需要多次查询，读放大严重。

---

## E

### 12. E2E 加密（端到端加密）

**英文**：End-to-End Encryption

**一句话解释**：消息在发送端加密，仅在接收端解密，服务端无法读取明文。

**详细说明**：Signal、WhatsApp 等隐私优先的 IM 采用端到端加密。 Signal Protocol 使用 X3DH 密钥协商 + Double Ratchet 算法，确保前向安全性（Forward Secrecy）和后向安全性。即使服务端被攻破，攻击者也无法解密历史消息。

### 13. Erlang

**英文**：Erlang

**一句话解释**：专为高并发、分布式、容错系统设计的函数式编程语言。

**详细说明**：WhatsApp 后端使用 Erlang 实现，仅用几十名工程师和数百台服务器支撑了数亿并发连接。Erlang 的 Actor 模型、热代码升级（Hot Code Swap）和"Let It Crash"哲学非常适合 IM 场景。

---

## F

### 14. FCM（Firebase Cloud Messaging）

**英文**：Firebase Cloud Messaging

**一句话解释**：Google 提供的跨平台推送服务，Android 设备的离线推送标准。

**详细说明**：FCM 是 Android 应用离线推送的事实标准。当 IM 应用在 Android 后台被杀进程后，服务端通过 FCM 向目标设备推送通知，实现离线消息提醒。

### 15. Fan-out（扩散）

**英文**：Fan-out

**一句话解释**：一条消息需要分发给多个接收者的过程。

**详细说明**：Fan-out 是群消息的核心操作。在写扩散模型中，发送一条群消息时会向每个群成员的收件箱各写一份（Fan-out on Write）；在读扩散模型中，各成员读取时各自拉取（Fan-out on Read）。

---

## G

### 16. Gateway（接入网关）

**英文**：Gateway

**一句话解释**：IM 系统中负责管理客户端长连接的入口层服务。

**详细说明**：接入网关负责维持海量客户端 TCP/WebSocket 长连接，处理连接的建立、心跳保活、消息转发等。常见的网关方案包括微信的 MM3、飞书的 LongLink、Telegram 的 Proxy 等。网关层是 IM 系统并发能力的关键。

### 17. Guild（Discord 服务器）

**英文**：Guild

**一句话解释**：Discord 中"服务器"的内部技术称呼，是一个社区组织单元。

**详细说明**：Discord 将用户社区称为 Guild，每个 Guild 包含多个频道（Channel）。Guild 架构涉及复杂的权限管理、Bot 集成和实时消息分发，是社区型 IM 的代表。

---

## H

### 18. Heartbeat（心跳机制）

**英文**：Heartbeat

**一句话解释**：客户端和服务端之间定期发送的小数据包，用于维持长连接和检测连接状态。

**详细说明**：移动网络环境下，NAT 超时会断开空闲连接。心跳包可以保持连接活跃，同时帮助客户端感知网络变化（如从 WiFi 切换到 4G）。微信 Mars 的智能心跳算法会根据网络环境动态调整心跳间隔，平衡保活效果和电量消耗。

---

## I

### 19. IM（即时通讯）

**英文**：Instant Messaging

**一句话解释**：允许用户实时收发文本、语音、视频等消息的通信系统。

**详细说明**：IM 系统涉及长连接管理、消息同步、存储、推送、安全等多个子系统。根据场景可分为社交 IM（微信、QQ）、企业 IM（飞书、钉钉）、隐私 IM（Signal）、社区 IM（Discord）等。

### 20. Idempotency（幂等性）

**英文**：Idempotency

**一句话解释**：同一操作执行多次产生相同结果，防止重复处理。

**详细说明**：在 IM 系统中，消息去重依赖幂等设计。客户端可能因网络重传导致同一条消息发送多次，服务端通过消息 ID 或客户端去重 ID 实现幂等处理，确保用户不会收到重复消息。

---

## J

### 21. Jitter（抖动）

**英文**：Jitter

**一句话解释**：网络延迟的波动变化。

**详细说明**：在 IM 弱网优化中，抖动是比平均延迟更影响体验的因素。Mars 等网络库通过自适应心跳间隔、连接复用、智能重连等策略应对网络抖动。

---

## K

### 22. KV 分离（Key-Value Separation）

**英文**：KV Separation / Tiered Storage

**一句话解释**：将 LSM-Tree 的索引与数据分离存储的优化技术。

**详细说明**：钉钉采用 KV 分离技术优化 LevelDB/RocksDB 存储引擎。将热数据（索引、元信息）放在高性能存储，冷数据（消息体）放在大容量存储，减少写放大并降低存储成本。WiscKey 论文是该技术的理论基础。

---

## L

### 23. Long Polling（长轮询）

**英文**：Long Polling

**一句话解释**：客户端发送请求后服务端保持连接，直到有新数据或超时才返回。

**详细说明**：长轮询是 WebSocket 普及前常见的实时通信方案。服务端 hold 住请求不立即返回，有新消息时立刻 push 给客户端。缺点是每次返回后需要重新建立连接，开销大于 WebSocket。

### 24. LSM-Tree

**英文**：Log-Structured Merge-Tree

**一句话解释**：一种面向写优化的数据结构，广泛用于 IM 系统存储引擎。

**详细说明**：LevelDB、RocksDB 等 KV 存储引擎基于 LSM-Tree。数据先写入内存 MemTable，达到阈值后 flush 为 SSTable 持久化到磁盘。通过 Compaction 合并层级，平衡读写性能。IM 消息的写密集特性使得 LSM-Tree 成为首选存储引擎。

---

## M

### 25. Mars（微信网络库）

**英文**：Mars

**一句话解释**：微信开源的跨平台客户端网络通信框架。

**详细说明**：Mars 是微信客户端的核心网络层，支持长连接管理、短连接请求、智能心跳、弱网优化等功能。其智能心跳算法能根据网络类型（WiFi/4G/3G）动态调整心跳间隔，在保活和省电之间取得平衡。目前已开源。

### 26. Mailbox Model（信箱模型）

**英文**：Mailbox Model

**一句话解释**：微信采用的消息同步模型，类似邮件系统。

**详细说明**：微信为每个用户维护一个收件箱，所有发送给该用户的消息都写入收件箱。客户端通过 SyncKey（最新同步位点）增量拉取收件箱中的新消息。信箱模型本质是读扩散——消息写一次到收件箱，读取时直接拉取。

### 27. MTProto（Telegram 协议）

**英文**：MTProto

**一句话解释**：Telegram 自研的应用层通信协议。

**详细说明**：MTProto 2.0 是 Telegram 的核心协议，设计目标是在不可靠网络环境下实现高效、安全的通信。它支持消息加密、断线重连后自动恢复、消息分块传输等特性。MTProto 不依赖 TLS，自行实现安全层，兼顾了 TCP 和 UDP 传输。

### 28. Multi-Device Sync（多端同步）

**英文**：Multi-Device Sync

**一句话解释**：同一用户在多个设备上保持消息一致性的机制。

**详细说明**：现代 IM 需要支持用户同时在手机、平板、电脑上使用。多端同步需要解决：消息一致到达各端、已读状态同步、消息操作（撤回、删除）多端同步等。飞书、Telegram 支持真正的多端实时同步，微信的多端体验也在持续优化。

---

## N

### 29. NAT Traversal（NAT 穿透）

**英文**：NAT Traversal

**一句话解释**：在 NAT（网络地址转换）环境下建立端到端连接的技术。

**详细说明**：移动设备通常位于 NAT 之后，NAT 设备会回收空闲连接的映射表项。IM 长连接需要通过心跳保持 NAT 映射活跃，否则连接会被静默断开。不同运营商和网络环境的 NAT 超时时间不同（通常 3-10 分钟）。

---

## P

### 30. PPM（推优先模型）

**英文**：Push Priority Model

**一句话解释**：钉钉采用的消息同步策略，优先推送，配合拉取兜底。

**详细说明**：PPM（Push Priority Model）是钉钉 DTIM 的核心同步策略。在线用户的消息直接通过长连接推送到客户端，延迟最低。对于不在线的用户或推送失败的情况，退化为拉取模式（Pull），客户端重连后通过 SyncKey 增量拉取。推优先模型兼顾了低延迟和可靠性。

### 31. PaxosStore

**英文**：PaxosStore

**一句话解释**：微信基于 Paxos 共识算法实现的分布式 KV 存储。

**详细说明**：PaxosStore 是微信存储层的核心组件，用于存储消息数据。它通过 Paxos 协议实现强一致性多副本写入，具备自动故障恢复、水平扩展能力。相比传统主从复制，PaxosStore 提供了更强的数据一致性和可用性保证。

### 32. Presence（在线状态）

**英文**：Presence

**一句话解释**：实时维护和广播用户在线/离线状态的服务。

**详细说明**：Presence 系统负责跟踪每个用户的在线状态（在线设备、网络类型、最后活跃时间等），并将状态变化通知给相关用户。QQ 的在线状态系统是其标志性功能之一。Presence 系统需要解决状态实时性、大规模状态广播的扇出问题。

---

## R

### 33. Read Receipt（已读回执）

**英文**：Read Receipt

**一句话解释**：告知消息发送方消息已被接收方阅读的功能。

**详细说明**：已读回执是 IM 的基本社交功能。设计上需要在消息粒度标记已读状态并多端同步。群消息已读回执更为复杂，需要聚合每个成员的已读位点。飞书支持精确到每条消息的群已读回执，微信仅在群聊中展示"XX等人已读"。

### 34. Roaming Message（消息漫游）

**英文**：Message Roaming

**一句话解释**：用户更换设备或重装应用后仍能查看历史消息的能力。

**详细说明**：消息漫游要求服务端持久化存储用户的历史消息。微信默认漫游 3-7 天，付费扩展更长时间。飞书、钉钉等企业 IM 通常提供更长的漫游周期。消息漫游需要在存储成本和用户体验之间取得平衡，通常采用冷热分离存储。

---

## S

### 35. Signal Protocol

**英文**：Signal Protocol

**一句话解释**：业界最知名的端到端加密协议，被 WhatsApp、Signal 等广泛采用。

**详细说明**：Signal Protocol 包含两个核心算法：X3DH（Extended Triple Diffie-Hellman）用于初始密钥协商，Double Ratchet（双棘轮）用于消息加密中的密钥轮换。双棘轮机制包含 DH 棘轮（每次消息生成新密钥对）和 Symmetric 棘轮（基于 HMAC 的密钥派生），确保前向安全和后向安全。

### 36. SyncKey（同步位点）

**英文**：SyncKey / Sequence Number

**一句话解释**：客户端记录的已同步消息的最新序列号，用于增量同步。

**详细说明**：微信的 SYNC 协议基于 SyncKey 实现增量同步。客户端本地保存一个 SyncKey，每次同步时将 SyncKey 发送给服务端，服务端返回该位点之后的所有新消息，同时更新 SyncKey。这种机制避免了全量同步的开销，是高效消息同步的基础。

### 37. ScyllaDB

**英文**：ScyllaDB

**一句话解释**：C++ 实现的高性能 NoSQL 数据库，兼容 Cassandra 接口。

**详细说明**：Discord 从 MongoDB 迁移到 ScyllaDB 来存储万亿条消息。ScyllaDB 采用 C++ 重写 Cassandra，利用 Shared-Nothing 架构和异步 I/O 实现 10 倍以上的性能提升。其分片（Shard）感知设计使得每条消息的读写固定路由到特定 CPU 核心，避免了锁竞争。

### 38. Smart Heartbeat（智能心跳）

**英文**：Smart Heartbeat

**一句话解释**：根据网络环境动态调整心跳间隔的算法。

**详细说明**：微信 Mars 的智能心跳算法会探测当前网络的 NAT 超时时间，选择略小于超时阈值的心跳间隔，最大化连接存活概率的同时最小化电量消耗。不同网络环境（WiFi/4G/3G/2G）使用不同的心跳策略，算法会自动学习和适应。

---

## T

### 39. Timeline Model（时间线模型）

**英文**：Timeline Model

**一句话解释**：以时间线为核心抽象的消息存储和同步模型。

**详细说明**：飞书采用 Timeline 模型，每个会话（单聊/群聊）维护一条有序的消息时间线。消息发送时追加到时间线末尾，客户端通过时间线的位点（Cursor）增量拉取新消息。Timeline 模型天然支持消息排序和范围查询，适合企业 IM 场景。

### 40. TCP vs UDP

**英文**：TCP vs UDP

**一句话解释**：两种核心传输层协议，IM 系统需要根据场景选择。

**详细说明**：QQ 早期选择 UDP 是因为 UDP 协议开销小、穿透性好、适合当时高延迟的网络环境。但 UDP 不保证可靠性，需要应用层实现重传和排序。随着网络条件改善，QQ 逐步迁移到 TCP。现代 IM 系统（微信、飞书、Telegram 等）普遍采用 TCP 或基于 TCP 的 WebSocket。

---

## W

### 41. WebSocket

**英文**：WebSocket

**一句话解释**：基于 TCP 的全双工通信协议，支持服务端主动推送。

**详细说明**：WebSocket 是 Web 端 IM 的标准长连接方案。它通过 HTTP Upgrade 握手建立连接后，转为全双工通信，服务端可以随时向客户端推送消息。相比 HTTP 长轮询，WebSocket 减少了连接建立和数据传输的开销。飞书 Web 端、Discord Web 端等均使用 WebSocket。

### 42. Write Diffusion（写扩散）

**英文**：Write Diffusion / Fan-out on Write

**一句话解释**：消息发送时同步写入每个接收者的收件箱。

**详细说明**：写扩散模式下，发送一条群消息时，系统会向每个群成员的收件箱各写一份副本。优点是读取时只需查询自己的收件箱，读性能好。缺点是写入放大严重——100人群发一条消息需要写 100 次。写扩散适合小群和读多写少的场景。

## X

### 43. X3DH（扩展三重 Diffie-Hellman）

**英文**：Extended Triple Diffie-Hellman

**一句话解释**：Signal Protocol 中用于初始会话密钥协商的异步协议。

**详细说明**：X3DH 允许 Alice 在 Bob 离线时，利用 Bob 预先上传到服务端的公钥（Identity Key + Signed PreKey + One-Time PreKey），计算出共享密钥，实现异步安全通信。这是端到端加密 IM 实现离线消息加密的基础。

## Z

### 44. 弱网优化

**英文**：Weak Network Optimization

**一句话解释**：针对网络质量差（高延迟、高丢包、频繁断连）环境的优化策略。

**详细说明**：弱网优化是移动端 IM 的核心挑战。常见策略包括：智能重连（指数退避）、消息压缩（Protobuf/自定义协议）、消息分片（大消息拆分传输）、请求合并（多条消息打包发送）、本地缓存（离线编辑后自动补发）等。微信 Mars、飞书 TTNet 等网络库都内置了完善的弱网优化能力。

### 45. 消息可靠性（Message Reliability）

**英文**：Message Reliability

**一句话解释**：确保消息不丢失、不重复、有序到达的设计目标。

**详细说明**：消息可靠性是 IM 系统的生命线。核心保障机制包括：客户端发送 → 服务端 ACK → 服务端持久化 → 推送目标客户端 → 客户端 ACK → 服务端标记已送达。任何环节失败都会触发重试。配合幂等去重和序列号排序，实现"至少一次"（At Least Once）的语义保证。
