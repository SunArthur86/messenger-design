# 消息系统设计：开发工程师视角

> 基于《飞书/微信/QQ 技术架构调研》提炼的设计方案
> 假设目标：千万级 DAU 的通用 IM 系统

---

## 一、核心设计决策

### 1.1 消息同步模型选型

| 方案 | 代表 | 优势 | 劣势 |
|------|------|------|------|
| 读扩散(Fan-out on Read) | 微信朋友圈 | 写入轻量 | 读放大，延迟高 |
| 写扩散(Fan-out on Write) | QQ 群消息 | 读取快 | 群大时写放大严重 |
| **Timeline 模型** | 飞书 | 多端同步天然支持，扩展性好 | 存储成本高 |

**推荐：Timeline 模型**

```
每个用户 → 1 条 Timeline（时间轴）
每条消息 → 写入发送方和所有接收方的 Timeline
读取时 → 从 Timeline 按位点拉取
```

理由：
- 天然支持多端同步（各端维护本地位点）
- 消息漫游简单（Timeline 保留历史）
- 已读回执自然实现（已读位点）
- 企业/社交场景都适用

### 1.2 消息 ID 设计

```
┌─────────────────────────────────────────────┐
│  Message ID (64 bit)                        │
├──────────┬──────────┬───────────────────────┤
│ 时间戳    │ 机器ID   │ 序列号                 │
│ 41 bit   │ 10 bit   │ 13 bit                │
│ (69年)   │ (1024台) │ (每ms 8192条)          │
└──────────┴──────────┴───────────────────────┘
```

- **服务端生成**：避免客户端时钟不一致
- **全局有序**：Snowflake 变体，按时间粗排序
- **用户级 sequence**：类似微信，每用户独立递增序列号用于同步

### 1.3 连接方案

```
                    ┌─────────────┐
                    │   Gateway   │
                    │  (网关层)    │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
        WebSocket      TCP 自定义    HTTP 短连接
        (主通道)       (备选)       (降级兜底)
```

**推荐：WebSocket 优先 + HTTP 降级**

| 策略 | 说明 |
|------|------|
| 主通道 | WebSocket 长连接，双向实时通信 |
| 心跳 | 客户端 30s 发一次，服务端 90s 超时 |
| 降级 | WebSocket 连接失败 → HTTPS 短连接轮询 |
| 重连 | 指数退避（1s→2s→4s→8s→30s 上限） |
| 多 IP | 本地缓存 → HTTP DNS → 系统 DNS |

## 三、核心模块详细设计

### 3.1 Timeline Store（消息存储）

```
表结构设计:

message (消息体，只存一份)
┌──────────┬──────────┬────────┬──────────┬────────┐
│ msg_id   │ conv_id  │ sender │ content  │ type   │
│ (PK)     │ (会话ID) │        │ (加密)   │        │
└──────────┴──────────┴────────┴──────────┴────────┘

timeline (每个用户每个会话一条时间轴)
┌──────────┬──────────┬──────────┬──────────┬──────────┐
│ user_id  │ conv_id  │ seq      │ msg_id   │ is_read  │
│ (PK)     │ (PK)     │ (递增)   │ (FK)     │          │
└──────────┴──────────┴──────────┴──────────┴──────────┘

conversation (会话)
┌──────────┬──────────┬──────────┬──────────┐
│ conv_id  │ type     │ members  │ max_seq  │
│ (PK)     │(单聊/群) │ (JSON)   │          │
└──────────┴──────────┴──────────┴──────────┘
```

**写扩散策略**：
- 单聊：写 2 条 Timeline（发送方 + 接收方）
- 群聊（< 500人）：写 N 条 Timeline（异步批量写入）
- 超大群（> 500人）：降级为读扩散，消息体只存一份

### 3.2 Sequence Service（序列号服务）

```go
// 类似微信 SeqSvr 的段分配策略
type SeqService struct {
    mu       sync.Mutex
    segments map[int64]*Segment // user_id → 当前段
}

type Segment struct {
    Current uint32 // 段内当前值
    End     uint32 // 段结束值
}

func (s *SeqService) Next(userID int64) uint32 {
    s.mu.Lock()
    defer s.mu.Unlock()
    
    seg := s.segments[userID]
    if seg == nil || seg.Current >= seg.End {
        // 从存储层分配新段（步长10000）
        seg = s.allocSegment(userID, 10000)
        s.segments[userID] = seg
    }
    seg.Current++
    return seg.Current
}
```

**关键设计**：
- 段分配：一次从 DB 取 10000 个号，内存中快速分配
- 持久化：段分配信息写 DB，重启不丢号
- 高可用：主备切换，备节点接手未用完的段

### 3.3 在线状态（Presence）

```
                    ┌─────────────┐
                    │  Redis      │
                    │  Hash:      │
   用户上线 ──────→ │  presence:  │ ──────→ 好友查询在线状态
   (WebSocket)      │  {uid:      │        (O(1) 查询)
                    │   status,   │
                    │   platform, │
                    │   last_seen │
                    │  }          │
                    └─────────────┘
                    
用户上线  → SET presence:{uid} {status:"online", platform:"ios", ts:1717400000}
用户下线  → SET presence:{uid} {status:"offline", ts:1717400000}
心跳超时  → 自动标记 offline（Redis Key 过期 + 后台扫描）
好友订阅 → 好友上线/下线时通过 WebSocket 推送通知
```

### 3.4 已读回执

```
消息已读流程:

1. 用户打开会话 → 读取到第 N 条消息
2. 客户端上报: {conv_id: "xxx", read_seq: N}
3. 服务端更新: timeline SET is_read=1 WHERE seq <= N AND user_id=ME
4. 通知发送方: 对方已读到第 N 条（WebSocket 推送）
5. 群聊: 聚合展示"X人已读"→ 批量查询 timeline 的 is_read 字段
```

## 四、关键技术方案

### 4.1 消息可靠性保证

```
发送可靠（不丢）:
┌──────┐     ┌──────┐     ┌──────┐
│Client│────→│Server│────→│Store │
│ [等待]│     │      │     │      │
│ [确认]│←────│ [ACK] │     │      │
└──────┘     └──────┘     └──────┘

接收可靠（不漏）:
- 在线: WebSocket 推送 + 客户端 ACK
- 离线: 上线后 sequence 差值拉取
- 兜底: 客户端定期（5分钟）主动检查 sequence 是否落后
```

### 4.2 消息去重

```
客户端本地去重:
- 发送: 本地生成 client_msg_id (UUID)
- 服务端: 检查 client_msg_id 是否已存在
- 接收: 本地 DB 按 msg_id 去重展示
```

### 4.3 群消息优化

| 群规模 | 策略 | 说明 |
|--------|------|------|
| 1-200人 | 写扩散 | 每人一条 Timeline，异步批量写入 |
| 200-2000人 | 写扩散 + 批量合并 | 批量 INSERT，消息体去重 |
| 2000+人 | 读扩散 | 消息只写一份，接收方按需拉取 |

### 4.4 弱网优化

```
策略一: 消息合并推送
  弱网下积压 10 条消息 → 合并为 1 次推送（减少 RTT）

策略二: 增量同步分页
  拉取 100 条未读 → 每页 20 条，逐页加载

策略三: 消息体压缩
  Protobuf 二进制编码（比 JSON 小 3-10 倍）

策略四: 图片/文件懒加载
  先推缩略图 URL，用户点击后才下载原图
```

### 4.5 消息加密

```
传输层: WSS (WebSocket over TLS)
存储层: AES-256-GCM 加密消息体
密钥管理: 每会话独立密钥，用户级密钥加密会话密钥

单聊 E2E 可选方案:
  双方 Diffie-Hellman 协商共享密钥
  → 用共享密钥加密消息
  → 服务端只看到密文（可选实现）
```

## 五、容量规划

### 5.1 千万 DAU 资源估算

| 模块 | QPS 峰值 | 存储 | 方案 |
|------|---------|------|------|
| Gateway | 500万 | - | 100 台 4C8G |
| MsgService | 100万 | - | 50 台 8C16G |
| TimelineStore | 200万读/50万写 | 50TB/年 | MySQL 分库分表 + Redis 缓存 |
| SeqService | 50万 | <1GB | 10 台 4C8G（内存分配） |
| PushService | 200万 | - | 20 台 4C8G |
| Presence (Redis) | 1000万 | 100GB | Redis Cluster 10 节点 |

### 5.2 分库分表策略

```
timeline 表按 user_id 分库分表:
- 256 库 × 16 表 = 4096 张表
- 分片键: user_id % 256 → 库, (user_id / 256) % 16 → 表
- 每张表预估 500 万行 → 单表 < 2GB

message 表按 conv_id 分库:
- 64 库
- 分片键: hash(conv_id) % 64
```

## 七、面试高频追问

1. **为什么选 Timeline 不选信箱模型？** → Timeline 天然支持多端同步和消息漫游，信箱模型需额外机制
2. **群 5000 人消息怎么处理？** → 写扩散太慢（5000次写），降级为读扩散
3. **消息如何保证不丢？** → 三重保障：推送+ACK、sequence 差值拉取、定期主动检查
4. **消息如何保证不重？** → client_msg_id 去重 + msg_id 去重
5. **10 亿用户如何存储？** → 分库分表 + 冷热分离（30天前归档到对象存储）
6. **如何处理消息乱序？** → 服务端单线程分配 sequence + 客户端按 seq 排序展示
